### Recording-specific: envelope encryption is **in-row**, not sidecar (added 2026-05-09)
The original Plan 12 design brief assumed a sidecar `.dek.enc` S3 object next to `.opus.enc`. **Plan 12.1 chose the simpler in-row strategy** ‚Äî `call_recordings.encrypted_dek bytea` lives in the same row as the metadata. The `dek_object_key` column (nullable) is reserved for v2 client-side decrypt scenarios that may want the DEK fetchable from S3 without a DB round-trip. For v1 playback (Plan 12.3) the service does `SELECT encrypted_dek FROM call_recordings ‚Üí KMS.Decrypt(dek) ‚Üí DEK plaintext ‚Üí AES-GCM-decrypt S3 object stream` ‚Äî one DB read + one KMS round-trip + one S3 GET, no sidecar object.

### Recording-specific: gRPC server uses `pool.WithTenant`, NOT `BypassRLS` (added 2026-05-09)
`tenancy_admin` (the role activated by `BypassRLS`) lacks SELECT/INSERT grants on `calls` and `call_recordings` ‚Äî those tables are owned by the `app` user and protected by RLS. The recording service therefore uses `pool.WithTenant(ctx, tenantID, fn)` for INSERT + audit + outbox in the same Tx. **`event_outbox` is owned by `tenancy_admin` BUT app retains full CRUD grants (Plan 03 setup)** so `outbox.Writer.Append` works inside a `WithTenant` Tx without role-switching. Plan 12.1 Task 3 verified this against migrations 000001 + 000002 + 000009.

### Recording-specific: gRPC mTLS identity is the source of truth (added 2026-05-09)
The gRPC `CommitRequest.tenant_id` and `CommitRequest.ingest_agent_id` wire fields are documentary. The handler **OVERRIDES `IngestAgentID` from the verified leaf cert's SPIFFE URI** before passing to the service, and rejects requests where `req.tenant_id != cert.tenant_id` with `codes.PermissionDenied`. SPIFFE URI shape: `spiffe://sociopulse/ingest-agent/<agent-id>?tenant=<tenant-uuid>`. This means a misconfigured ingest-uploader cannot forge attribution.

### Recording-specific: AAD = `tenant_id.String()` for envelope encryption (added 2026-05-09)
Plan 12.2 locked in `aad := []byte(row.TenantID.String())` as the **AEAD additional-data** binding for both the KMS DEK unwrap step AND the AES-256-GCM audio decrypt step. Plan 08 (FreeSWITCH cluster ‚Äî when it ships) MUST encrypt with the same AAD so playback can verify the binding. Rationale: copy-paste defence ‚Äî encrypted blobs from tenant A cannot be decrypted against tenant B's row, even if an attacker swaps `audio_object_key` in the DB. CallID is intentionally NOT in the AAD: at commit time the ingest-uploader knows tenant + call but the recording row's `id` is server-generated.

### Recording-specific: Local* ‚Üí Yandex SDK migration path (added 2026-05-09)
Plan 12.2 ships `crypto.LocalDEKUnwrapper` and `storage.LocalObjectStore` as the production implementations of the recording-internal ports. Plan 01 (Yandex Cloud infra) will add `-tags=yandex_kms` / `-tags=yandex_s3` build-tag-gated files (`crypto/yandex_kms.go`, `storage/yandex_s3.go`) that satisfy the same interfaces but call the Yandex SDK. Until then, production runs with the Local* ports ‚Äî fine for staging/dev, not for compliance prod (the LocalDEKUnwrapper holds the KEK plaintext in memory rather than offloading to KMS HSM). The `recording.LocalKEKs` config field is the bridge: empty in prod-with-yandex-tag (the Yandex impl ignores it); populated with `kek-platform-test: "<64 hex chars>"` for staging.

### Recording-specific: 200 MiB in-RAM playback cap is v1 (added 2026-05-09)
`crypto.AudioDecryptor.Decrypt` reads the entire encrypted audio into memory, validates the AES-GCM auth tag, then returns a `bytes.Reader` to the HTTP layer. The 200 MiB cap is enforced BEFORE allocation (peeking `size` arg first). v2 will introduce a chunked-envelope format with per-chunk auth tags so playback can stream-validate without buffering ‚Äî Plan 08 owns the encrypt-pipeline change and Plan 12.x will add a parallel `ChunkedAudioDecryptor` impl. Plan 12.3 sets `Accept-Ranges: none` on the HTTP response so clients don't send Range headers (which AES-GCM single-tag format can't satisfy without the buffer).

### Recording-specific: workers use `BypassRLS` for cross-tenant sweeps (added 2026-05-10)
Plan 12.4 retention + integrity workers run cross-tenant ‚Äî there is no per-request tenant context. The `internal/recording/store/lifecycle.go` methods (ListDueColdMoves / ListDueDeletes / SampleForVerify / MarkCold / MarkDeleted / UpdateVerifyResult) all run inside `pool.BypassRLS` (mirrors `internal/dialer/retry.PgReader`). The `*Tx` variants (`MarkColdTx`, `MarkDeletedTx`, `UpdateVerifyResultTx`) take the caller's already-tenant-scoped Tx for atomic audit + status-flip writes inside `pool.WithTenant(row.TenantID, ...)`. Plan 12.1 originally claimed `tenancy_admin` had grants on `call_recordings` ‚Äî it did not. Plan 12.4 added migration `000011_admin_grants_call_recordings` (SELECT + UPDATE only, no DELETE ‚Äî retention flips status='deleted' not physical-delete). Future cross-tenant scans on any `call_recordings`-adjacent table need the same audit + grant migration.

### Recording-specific: leader-elected daemons share dialer's leader primitive (added 2026-05-10)
Plan 12.4 reuses `internal/dialer/retry.PgLeader` directly ‚Äî it accepts a custom int64 lock key. Recording lock keys are exported as `worker.RetentionLockKey = fnvHash("recording.retention_pass")` and `worker.IntegrityLockKey = fnvHash("recording.integrity_pass")` ‚Äî distinct FNV-1a hashes from the dialer's `fnvHash("dialer.retry")` so the three daemons never contend on the same advisory lock slot. If a fourth worker shows up wanting the same primitive, time to extract `PgLeader` to `pkg/postgres/leader`. Until then, the cross-package import is acceptable duplication.

### Recording-specific: integrity worker's "weekly 1% sample" is stateless (added 2026-05-10)
No `worker_runs` table needed for scheduling. The eligibility filter `verified_at IS NULL OR verified_at < now() - interval '7 days'` is the rate-limiter ‚Äî rows verified within the past 7 days are deliberately skipped. With default `Interval=1h`, `Batch=10`, `SamplePercent=1.0` (BERNOULLI), the math is 168 ticks √ó 10 rows = 1680 rows verified/week. For a 36.5M-row table that's well under the spec's 1% target ‚Äî operators tune the three knobs in `recording.workers` config to scale the sample budget.

### Billing-specific: audit-on-tariff-PATCH is best-effort post-Update (added 2026-05-15)
`TariffStore.Update` opens its own Tx via `UpsertSettings` ‚Äî no path to inject an external Tx in v1. `AuditEmitter.EmitTariffUpdated` runs in its own `pool.WithTenant` Tx AFTER the tariff write commits; failure logs WARN, returns nil (response is already 200). At-most-once audit on tariff change is the explicit trade-off. To upgrade to strict atomicity (152-–§–ó auditor scenario): refactor `TariffStore.Update` to take `*postgres.Tx` so the handler can compose tariff update + outbox append in one Tx. Tracked in Plan 14 Production lessons.

### Billing-specific: NO audit per call_costs INSERT (added 2026-05-15)
At hundreds of calls/minute, an audit row per `OnCallFinalized` would dwarf the actual audit signal. Audit only fires on the human-driven `tenant.<t>.audit.event` with `Action="billing.tariff_updated"` ‚Äî version_before / version_after / changed_keys[]. The numeric tariff values live in `tenant_settings.billing.*` and are queryable out-of-band.

### Billing-specific: day-31 month-arithmetic gotcha (added 2026-05-15)
Go's `time.AddDate(0, -1, 0)` calendar-normalises overflow days: Jan 31 minus 1 month ‚Üí Mar 3 (since Feb has 28 days), silently skipping February. `SpendByMonth` originally iterated from `time.Now()` and would have produced bogus monthly series for any caller on day 29/30/31. Fix: snap the anchor to `time.Date(year, month, 1, ...)` BEFORE iterating. Regression test pinned at Jan 31 + Mar 31. Apply this pattern anywhere you do `prev_month := now.AddDate(0, -1, 0)`.

### Billing-specific: `eventbus.Subscriber` is the canonical NATS abstraction (reinforced 2026-05-15)
Never use raw `nats.go`/`jetstream` directly in modules. Pattern: `bus.Subscribe(ctx, wildcardSubject, queueGroup, handler)`. Handler signature `func(subject string, payload []byte) error` carries no ctx ‚Äî use `context.Background()` for the hook invocation (NATS delivery has its own lifecycle independent of any HTTP request). When a future refactor adds ctx to the handler signature, propagate it via `context.WithoutCancel` for durable writes.

### Billing-specific: `auth` module not in cmd/api registry (added 2026-05-15)
Discovered during Plan 14 Step I: `cmd/api/main.go` does NOT register `auth.Module{}`, so `auth.RBACChecker` is never published to the locator. Billing's `requireRBAC` correctly degrades ‚Äî admin/supervisor JWT-role fast-path short-circuits; the matrix-checker fallback is skipped (`rbac_wired=false`). Tracked as separate follow-up ‚Äî does NOT block any current plan, but a future auth-mod-related plan should fix this so the RBAC matrix is reachable.

### `cmd/api/TestRunStartsAndShutsDownCleanly` is pre-existing macOS flake (added 2026-05-15)
Verified by reverting Plan 14 to `2a33ea7` ‚Äî same failure. Environment-specific port-binding timing on macOS; CI on Linux is consistently green (last 8 successful CI runs on main). Do NOT attempt to fix as part of a feature plan ‚Äî orthogonal issue. Track separately.
